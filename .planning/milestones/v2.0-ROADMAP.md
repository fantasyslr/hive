# Milestone v2.0: Intelligence Layer

**Status:** SHIPPED 2026-04-01
**Phases:** 4-7
**Total Plans:** 12

## Overview

Transform Hive from a passive dispatch board into an active intelligence platform. Upgraded worker from shell wrapper to typed runtime, added structured memory extraction with history injection, enabled complex task decomposition with coordinator pattern, and wired everything with declarative hooks.

## Phases

### Phase 4: Worker Runtime Foundation
**Goal**: Workers are typed runtime clients with self-describing capabilities and structured output — not shell wrappers calling CLI commands
**Depends on**: Phase 3 (stable task system and memory infrastructure)
**Plans**: 4 plans

Plans:
- [x] 04-01: Package scaffold + type contracts (HarnessAdapter, StructuredResult, ToolDefinition) + AgentCard extension
- [x] 04-02: Three harness adapters (Claude, Gemini, Codex) + extractStructuredResult utility
- [x] 04-03: Two-layer tool registry + gateway AgentRegistry harness metadata
- [x] 04-04: Worker bridge rewrite + delete bash adapter

### Phase 5: Structured Memory + History Injection
**Goal**: Every completed task produces structured knowledge that future tasks automatically receive as context
**Depends on**: Phase 4 (structured result contract provides clean input for LLM extraction)
**Plans**: 3 plans

Plans:
- [x] 05-01: Type contracts (MemoryConclusion extension, HistoryContext) + extractWithLLM + LlmClient
- [x] 05-02: Wire LLM extraction into MemoryService.writeConclusion with structured storage
- [x] 05-03: HistoryInjector with dual-channel retrieval + wire into Dispatcher.autoAssign

### Phase 6: Coordinator + Session Mode
**Goal**: Complex goals decompose into ordered sub-task DAGs that execute and synthesize results automatically
**Depends on**: Phase 5 (coordinator benefits from history injection on sub-tasks)
**Plans**: 3 plans

Plans:
- [x] 06-01: Extend TaskKind with coordinate/synthesize + POST /tasks/batch API
- [x] 06-02: Persistent worker session in HarnessAdapter + ClaudeAdapter
- [x] 06-03: CoordinatorService LLM decomposition + synthesis trigger in DependencyUnblocker

### Phase 7: Hook Engine
**Goal**: Lifecycle behaviors defined declaratively in JSON config — the team adds automations without touching code
**Depends on**: Phase 6 (hook engine wires up all prior features)
**Plans**: 2 plans

Plans:
- [x] 07-01: HookEngine core — Zod schema, condition evaluator, ActionHandler interface
- [x] 07-02: Three action handlers (http, create_task, memory_search) + gateway wiring + fs.watch hot-reload

## Milestone Summary

**Key Decisions:**
- Worker Runtime first — typed adapter + tool registry + structured results are the foundation
- SMEM+HINJ after runtime — structured result contract feeds LLM extraction
- CORD+Session after SMEM — coordinator benefits from history injection
- HOOK last — connects everything with declarative events, needs stable substrate
- CC source deep dive (freestylefly/claude-code) revealed worker runtime is prerequisite

**Issues Resolved:**
- Gateway wiring gap — LlmClient, HistoryInjector not connected in index.ts (fixed in audit gap closure)
- worker-adapter.sh replaced by typed TypeScript adapters (one-step migration)

**Technical Debt:**
- ToolRegistry created but not consumed by worker-bridge or gateway at runtime (orphaned service)
- 3 pre-existing TS errors in feishu-webhook.ts and templates.ts (from v1.0 phases)
- LlmClient requires ANTHROPIC_API_KEY env var — services gracefully degrade to null when absent
